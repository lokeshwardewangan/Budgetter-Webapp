# Security Policy

## Reporting a Vulnerability

If you discover a security vulnerability in Budgetter, please report it privately. **Do not open a public GitHub issue.**

To report a vulnerability, email:

**lokeshwar.prasad.cse@gmail.com**

Please include:

- A clear description of the issue
- Steps to reproduce
- Affected version, component, or endpoint
- Any potential impact you have identified

## Response Time

- **Acknowledgement:** within 72 hours of report
- **Initial assessment:** within 7 days
- **Fix or mitigation plan:** communicated as soon as the issue is validated

We will keep you informed throughout the process and credit you in the fix notes if you wish.

## What Qualifies as a Security Issue

Examples of issues we consider security-relevant:

- Authentication or authorization bypass
- Token, session, or password handling flaws
- Sensitive data exposure (PII, credentials, secrets)
- SQL injection, XSS, CSRF, SSRF, or remote code execution
- Insecure direct object references (IDOR)
- Vulnerabilities in dependencies that directly affect Budgetter

The following are generally **not** considered security issues:

- Missing best-practice HTTP headers without a demonstrated impact
- Reports generated solely by automated scanners without proof of exploitability
- Issues requiring physical access or already-compromised accounts
- Self-XSS or social engineering

Thank you for helping keep Budgetter and its users safe.
